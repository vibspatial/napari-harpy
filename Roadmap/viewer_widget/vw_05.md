# VW-05 Implementation Notes

## Recommendation

Start `VW-05` by wiring `ObjectClassificationWidget` to shared app state
before changing its live-label annotation flow too aggressively.

The current codebase already gives us a strong foundation for this migration:

- `HarpyAppState` already exposes the shared loaded `sdata`
- `HarpyAppState` already emits `sdata_changed`
- `ViewerAdapter` already knows how to:
  - load labels layers from `sdata`
  - reuse already loaded labels layers
  - activate a requested labels layer
- `FeatureExtractionWidget` has already completed the shared-`sdata`
  migration, so we now have a working reference pattern
- the object-classification controller stack is already fairly modular:
  - `AnnotationController`
  - `ClassifierController`
  - `ViewerStylingController`
  - `PersistenceController`
- table and feature-matrix discovery are already implemented through:
  - `get_annotating_table_names(...)`
  - `get_table_metadata(...)`
  - `get_table_obsm_keys(...)`
  - `validate_table_binding(...)`

That means `VW-05` is mainly a migration and orchestration task:

- replace viewer scanning as the source of selectable segmentation options
- subscribe the widget to shared `sdata`
- make the selection flow coordinate-system-first
- auto-load and activate the selected labels layer through `ViewerAdapter`
- keep annotation dependent on a live labels layer
- finish the remaining legacy UX cleanup

UI rule for `VW-05`:

- the widget should behave like a shared-state consumer
- users choose the coordinate system first
- segmentation, table, and feature-matrix options should then be derived from
  that coordinate-system context in the loaded `sdata`
- annotation should still only become active once a compatible labels layer is
  actually loaded and bound in napari

## Current State Compared To The VW-05 Ticket

### Already available

- shared `HarpyAppState` per viewer
- shared loaded `sdata`
- `sdata_changed` signal
- `ViewerAdapter.ensure_labels_loaded(...)`
- `ViewerAdapter.activate_layer(...)`
- `ViewerAdapter.get_loaded_labels_layer(...)`
- `ClassifierController` with stable bind / retrain flow
- `PersistenceController` with write / reload flow
- `AnnotationController` that already keeps annotation dependent on a live
  labels layer
- `ViewerStylingController` that already styles the live labels layer from the
  selected table
- pure helper logic for:
  - annotating tables
  - table metadata
  - feature-matrix keys
  - table-binding validation

### Still legacy / not yet aligned with VW-05

- the UI still has a `Rescan Viewer` button
- the widget still carries a small refresh-button path that reads like legacy
  viewer-rescan UX even though shared `sdata` is now the main source of truth
- labels-layer lifecycle is now handled through `ViewerAdapter` signals rather
  than raw viewer scanning, but the remaining UX cleanup is still unfinished

### Architectural takeaway

The object-classification controller layer is already reasonably well
factored.

The main missing work is no longer the core migration itself. The shared-`sdata`
selection flow, live labels-layer preparation, and controller rebinding are
already in place. What remains is mainly:

- confirming the last table / classifier / persistence alignment details
- removing the final `Rescan Viewer` semantics
- polishing the remaining shared-state messaging and transitional UX

So `VW-05` should focus on:

- finishing the remaining table / classifier / persistence alignment checks
- removing the final `Rescan Viewer` semantics
- polishing the last shared-state messaging so the widget reads like a Harpy
  state consumer rather than a transitional viewer tool

not on reworking the underlying annotation or classifier logic from scratch.

The target UI should therefore be:

1. choose coordinate system
2. choose segmentation mask
3. auto-load / activate the corresponding labels layer in that coordinate system
4. choose linked table
5. choose feature matrix
6. annotate, retrain, color, write, and reload against that live bound layer

Status summary at the current codebase state:

- Slice 4: finished
- Slice 5: basically finished
- Slice 6: partially started, but not finished

## Reuse vs. Avoid

### Reuse

These pieces are already aligned with the target architecture:

- `HarpyAppState`
- `sdata_changed`
- `ViewerAdapter`
- `ClassifierController`
- `PersistenceController`
- `AnnotationController` annotation semantics
- `ViewerStylingController` styling semantics
- `get_annotating_table_names(...)`
- `get_table_metadata(...)`
- `get_table_obsm_keys(...)`
- `validate_table_binding(...)`
- shared labels option models from `_spatialdata.py`

### Avoid as the basis for the new object-classification flow

These are still tied to older viewer-scanning or transitional assumptions:

- `Rescan Viewer` as the primary way to discover or refresh segmentations
- refresh-button flows that imply the widget needs a manual viewer rescan
- raw viewer-layer events as the primary widget refresh contract instead of
  shared app state plus `ViewerAdapter` signals

Those may remain temporarily during the migration, but they should not remain
the main source of truth once `VW-05` is complete.

### Slice 7

Close the remaining integration gap between `Feature Extraction` and
`Object Classification`.

Status: not started

Problem to solve:

- when `Feature Extraction` writes a feature matrix into
  `.obsm[feature_key]`, `Object Classification` does not currently receive a
  semantic update about that table change
- as a result:
  - newly written feature-matrix keys may not appear automatically in the
    object-classification feature-matrix selector
  - if the currently selected feature matrix was overwritten in place, the
    classifier is not automatically marked stale
  - the classifier does not retrain automatically, but it also does not know
    that its predictions may now be out of date

Recommended implementation:

- add a semantic shared-state signal in `HarpyAppState`, for example
  `table_state_changed`
- emit a structured payload rather than a generic refresh signal, for example:
  - selected `sdata`
  - `table_name`
  - `label_name`
  - changed `feature_key`
  - optional source, e.g. `feature_extraction`
- have `FeatureExtractionWidget` / `FeatureExtractionController` emit that
  signal when a feature-extraction job successfully writes into the selected
  table
- have `ObjectClassificationWidget` subscribe to that semantic table-change
  signal and:
  - ignore updates for other `sdata` objects or other tables
  - refresh feature-matrix keys when the selected table changed
  - preserve the current selected feature key when still valid
  - mark the classifier stale only when the changed feature key matches the
    currently selected feature matrix

Recommended behavior:

- same-session Harpy edits to the shared in-memory table should propagate
  immediately across widgets
- new feature-matrix keys written by `Feature Extraction` should appear in the
  object-classification feature-matrix selector without any manual rescan
- if the currently selected feature matrix is overwritten in place, the
  classifier should become stale
- retraining should remain an explicit user action rather than happening
  automatically in the background
- the existing `Reload` action should remain the explicit way to discard the
  current in-memory table state and re-read the table from disk after external
  changes (for example from a notebook or other code)

Why this is the cleanest path:

- `sdata_changed` is too broad for this case because the `SpatialData` object
  identity does not change when `.obsm[...]` is updated in place
- a semantic `table_state_changed` signal avoids over-refreshing the whole
  widget tree
- it lets `Object Classification` react precisely to the table / feature-matrix
  changes that matter, instead of treating all `sdata` updates as equivalent
- it keeps the mental model clean:
  - shared in-memory Harpy session state updates live
  - `Reload` is for explicit disk resynchronization, not for cross-widget sync

Recommended tests:

- after feature extraction writes a new feature key, the object-classification
  feature-matrix selector refreshes automatically
- when the currently selected feature key is overwritten, the classifier is
  marked stale
- when a different feature key is written, the classifier selection remains
  stable and is not marked stale unnecessarily
- no automatic retrain is triggered by feature extraction alone
