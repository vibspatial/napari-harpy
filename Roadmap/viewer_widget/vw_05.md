# VW-05 Implementation Notes

## Recommendation

Start `VW-05` by wiring `ObjectClassificationWidget` to shared app state
before changing its live-label annotation flow too aggressively.

The current codebase already gives us a strong foundation for this migration:

- `HarpyAppState` already exposes the shared loaded `sdata`
- `HarpyAppState` already emits `sdata_changed`
- `ViewerAdapter` already knows how to:
  - load labels layers from `sdata`
  - reuse already loaded labels layers
  - activate a requested labels layer
- `FeatureExtractionWidget` has already completed the shared-`sdata`
  migration, so we now have a working reference pattern
- the object-classification controller stack is already fairly modular:
  - `AnnotationController`
  - `ClassifierController`
  - `ViewerStylingController`
  - `PersistenceController`
- table and feature-matrix discovery are already implemented through:
  - `get_annotating_table_names(...)`
  - `get_table_metadata(...)`
  - `get_table_obsm_keys(...)`
  - `validate_table_binding(...)`

That means `VW-05` is mainly a migration and orchestration task:

- replace viewer scanning as the source of selectable segmentation options
- subscribe the widget to shared `sdata`
- make the selection flow coordinate-system-first
- auto-load and activate the selected labels layer through `ViewerAdapter`
- keep annotation dependent on a live labels layer
- finish the remaining legacy UX cleanup

UI rule for `VW-05`:

- the widget should behave like a shared-state consumer
- users choose the coordinate system first
- segmentation, table, and feature-matrix options should then be derived from
  that coordinate-system context in the loaded `sdata`
- annotation should still only become active once a compatible labels layer is
  actually loaded and bound in napari

## Current State Compared To The VW-05 Ticket

### Already available

- shared `HarpyAppState` per viewer
- shared loaded `sdata`
- `sdata_changed` signal
- `ViewerAdapter.ensure_labels_loaded(...)`
- `ViewerAdapter.activate_layer(...)`
- `ViewerAdapter.get_loaded_labels_layer(...)`
- `ClassifierController` with stable bind / retrain flow
- `PersistenceController` with write / reload flow
- `AnnotationController` that already keeps annotation dependent on a live
  labels layer
- `ViewerStylingController` that already styles the live labels layer from the
  selected table
- pure helper logic for:
  - annotating tables
  - table metadata
  - feature-matrix keys
  - table-binding validation

### Still legacy / not yet aligned with VW-05

- `ObjectClassificationWidget` still uses `SpatialDataViewerBinding`
- the widget still refreshes from viewer layer events
- the UI still has a `Rescan Viewer` button
- `AnnotationController` and `ViewerStylingController` still resolve the live
  labels layer through `SpatialDataViewerBinding`

### Architectural takeaway

The object-classification controller layer is already reasonably well
factored.

The main missing work is migration of option discovery and live labels-layer
binding from viewer scanning to shared `sdata` plus `ViewerAdapter`.

So `VW-05` should focus on:

- ensuring selected labels are loaded and activated automatically
- keeping annotation/styling bound to a real live labels layer
- removing the final `Rescan Viewer` semantics

not on reworking the underlying annotation or classifier logic from scratch.

The target UI should therefore be:

1. choose coordinate system
2. choose segmentation mask
3. auto-load / activate the corresponding labels layer in that coordinate system
4. choose linked table
5. choose feature matrix
6. annotate, retrain, color, write, and reload against that live bound layer

## Reuse vs. Avoid

### Reuse

These pieces are already aligned with the target architecture:

- `HarpyAppState`
- `sdata_changed`
- `ViewerAdapter`
- `ClassifierController`
- `PersistenceController`
- `AnnotationController` annotation semantics
- `ViewerStylingController` styling semantics
- `get_annotating_table_names(...)`
- `get_table_metadata(...)`
- `get_table_obsm_keys(...)`
- `validate_table_binding(...)`
- shared labels option models from `_spatialdata.py`

### Avoid as the basis for the new object-classification flow

These are still tied to viewer scanning / legacy metadata assumptions:

- `SpatialDataViewerBinding`
- `refresh_segmentation_masks()` as the primary source of options
- `viewer.layers.events.inserted/removed/reordered` as the primary refresh path
- `Rescan Viewer` as the primary way to discover segmentations

Those may remain temporarily during the migration, but they should not remain
the main source of truth once `VW-05` is complete.

## Recommended Implementation Slices

### Slice 1

Wire the widget to shared app state and `sdata_changed`.

Status: completed

Implement:

- subscribe `ObjectClassificationWidget` to
  `self._app_state.sdata_changed.connect(...)`
- add a `refresh_from_sdata(sdata)` path
- when no `sdata` is loaded:
  - show a clear empty-state / status message
  - clear selector options
  - disable annotation / classifier / persistence actions
- seed the widget from `self._app_state.sdata` on construction

Keep for this slice:

- the existing controller stack
- the existing class editor / retrain / persistence UI
- the existing viewer-binding based live-layer resolution as a temporary
  compatibility path

Recommended tests:

- widget reflects `self._app_state.sdata` on construction
- widget refreshes when `sdata_changed` fires
- widget becomes empty/disabled when `sdata` is cleared

Delivered in this slice:

- `ObjectClassificationWidget` now subscribes to
  `self._app_state.sdata_changed`
- the widget now exposes `refresh_from_sdata(sdata)` and seeds itself from the
  shared app state on construction
- when shared `sdata` is `None`, the widget clears selector inputs, disables
  the rescan action, and shows a `No SpatialData Loaded` status card
- the legacy viewer-binding flow remains in place for discovering currently
  loaded live labels layers, but it is now gated behind the presence of shared
  `sdata`

### Slice 2

Replace viewer-scanned segmentation discovery with `sdata`-driven option
discovery and introduce a coordinate-system-first selector flow.

Status: completed

Implement:

- add explicit coordinate-system state to `ObjectClassificationWidget`
- expose coordinate systems from the loaded `sdata`
- replace labels discovery with a pure helper that reads labels options from
  shared `sdata`
- filter segmentation options by the selected coordinate system
- preserve stable selection where possible across refreshes
- keep:
  - table lookup through `get_annotating_table_names(...)`
  - table metadata through `get_table_metadata(...)`
  - feature-matrix lookup through `get_table_obsm_keys(...)`

Recommended behavior:

- coordinate-system selection should happen inside the widget
- segmentation options should now come from `self._app_state.sdata` and be
  filtered by the selected coordinate system
- viewer layer insertion/removal should no longer be required to discover
  selectable segmentations

Recommended tests:

- coordinate-system choices come from the loaded `sdata`
- segmentation choices come from the loaded `sdata`
- segmentation choices are filtered by the selected coordinate system
- linked tables come from the selected labels element in `sdata`
- feature matrices come from the selected linked table in `sdata`

Delivered in this slice:

- `ObjectClassificationWidget` now exposes an explicit coordinate-system
  selector fed from shared `self._app_state.sdata`
- segmentation options now come directly from shared `sdata` instead of
  `SpatialDataViewerBinding`
- segmentation discovery no longer depends on currently loaded viewer layers
- segmentation options are filtered by the selected coordinate system and keep
  a stable selection when possible across refreshes
- linked tables continue to come from `get_annotating_table_names(...)`
  against the selected labels element in `sdata`
- feature-matrix keys continue to come from `get_table_obsm_keys(...)` against
  the selected linked table in `sdata`
- the live labels-layer binding path remains unchanged for now, so the widget
  can already expose selections from `sdata` while still warning when the
  chosen segmentation is not yet loaded as a napari labels layer

### Slice 3

Auto-load and activate the selected labels layer through `ViewerAdapter`.

Status: completed

Implement:

- when a coordinate system and segmentation are selected:
  - resolve whether a matching labels layer is already loaded
  - if not, load it through `self._app_state.viewer_adapter.ensure_labels_loaded(...)`
  - activate it through `activate_layer(...)`
- use the explicitly selected coordinate system as the load target
- surface clear status messaging when the widget loads or activates the labels
  layer for the user

Recommended behavior:

- choosing a segmentation should prepare the live labels layer automatically
  in the selected coordinate system instead of requiring the user to preload
  it manually
- if loading fails, the widget should remain informative and keep annotation
  disabled

Recommended tests:

- choosing a segmentation loads a labels layer when needed
- repeated selection reuses the existing labels layer
- the selected coordinate system is respected
- the selected labels layer is activated in the viewer

Delivered in this slice:

- selecting a coordinate system plus segmentation now prepares the live labels
  layer automatically through `self._app_state.viewer_adapter`
- when a matching labels layer already exists, the widget reuses it, registers
  it with the shared layer bindings when needed, and activates it in the viewer
- when no matching labels layer exists, the widget loads it through
  `ensure_labels_loaded(...)` in the explicitly selected coordinate system and
  activates the resulting layer
- the object-classification status card now surfaces whether the widget loaded
  or activated the segmentation layer for the user
- transitional legacy labels layers without explicit coordinate-system metadata
  are still reused when they otherwise match the selected `SpatialData` plus
  labels element, which keeps the migration compatible with older tests and
  viewer state

### Slice 4

Migrate annotation and styling bindings off `SpatialDataViewerBinding`.

Status: completed

Implement:

- update `AnnotationController` to bind against the live labels layer through
  `ViewerAdapter` / shared layer bindings rather than viewer scanning
- update `ViewerStylingController` the same way
- keep annotation dependent on a real live labels layer
- keep styling tied to the selected labels layer and selected table

Recommended behavior:

- annotation should remain disabled until a compatible live labels layer is
  actually bound
- styling should continue to refresh the same live labels layer after
  annotation / classifier changes

Recommended tests:

- annotation controller binds to the auto-loaded live labels layer
- object pick / user class actions still work after auto-loading
- styling refresh continues to target the correct labels layer

Delivered in this slice:

- `AnnotationController` now binds live labels layers through the shared
  `ViewerAdapter` instead of `SpatialDataViewerBinding`
- `ViewerStylingController` now resolves the current labels layer through the
  same shared `ViewerAdapter`
- `ObjectClassificationWidget` no longer constructs or depends on
  `SpatialDataViewerBinding` for its runtime live-layer path
- the widget's labels-layer preparation step now asks `ViewerAdapter` for an
  already loaded matching labels layer before deciding whether to load one
- annotation and styling both stay attached to the same auto-loaded or reused
  live labels layer, which keeps object picking and color refresh aligned

### Slice 5

Keep table, feature-matrix, and classifier flows aligned with shared `sdata`.

Implement:

- ensure table-binding validation still gates the effective table binding
- keep feature-matrix selection derived from the selected linked table
- keep classifier binding / stale-state behavior consistent after the migration
- ensure persistence write / reload still operate on the shared in-memory
  `sdata` and selected table context

Recommended behavior:

- table / feature-matrix choices should remain driven by the selected labels
  element in shared `sdata`
- retrain, write, and reload should remain disabled whenever the effective
  classification context is incomplete

Recommended tests:

- invalid table bindings still disable dependent actions
- feature-matrix choices refresh from the selected linked table
- retrain / write / reload states remain correct after selection changes

### Slice 6

Clean up legacy viewer-scanning UX and finalize status messaging.

Implement:

- remove or repurpose `Rescan Viewer`
- remove now-obsolete viewer event refresh semantics from the primary path
- update selection/status text so it explains the shared-`sdata` plus
  auto-load workflow
- keep annotation/classifier/persistence feedback consistent with the new
  source of truth

Recommended end-state:

- the object-classification widget should read like a shared-state consumer,
  not a viewer-scanning tool
- any remaining viewer-scanning code should be only transitional or removed

Recommended tests:

- no user action depends on `Rescan Viewer`
- the widget stays synchronized through `sdata_changed`
- selection status remains informative after the migration

## Recommended First Slice To Implement

Start with:

1. app-state wiring
2. `sdata_changed` subscription
3. empty-state refresh path

- it is the smallest migration step with visible value
- it does not yet force a rewrite of the live labels-layer annotation flow

  resolution
