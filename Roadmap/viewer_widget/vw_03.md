# VW-03 Implementation Notes

## Recommendation

Do not start `VW-03` with the file dialog.

The current codebase already gives us a good foundation for the `Viewer`
widget:

- `Interactive(sdata)` can already inject a loaded `SpatialData` object into
  `HarpyAppState`
- `HarpyAppState` already exposes `sdata_changed`
- `ViewerAdapter` already supports:
  - `ensure_labels_loaded(...)`
  - `ensure_image_loaded(..., mode="stack")`
  - `ensure_image_loaded(..., mode="overlay")`
  - `activate_layer(...)`

That means we can build most of the `Viewer` widget in a state-driven way first
and add the `Open SpatialData` file-picker UX later.

## Current State Compared To The VW-03 Ticket

### Already available

- shared `HarpyAppState` per viewer
- shared loaded `sdata`
- `sdata_changed` signal
- `ViewerAdapter` and `LayerBindingRegistry`
- real loading paths for:
  - labels
  - stack images
  - overlay images
- `Interactive(sdata)` programmatic launch path

### Not yet implemented

- no `Viewer` widget file
- no `Viewer` widget command in `napari.yaml`
- `Interactive` does not yet open a `Viewer` dock widget
- no widget subscribed to `sdata_changed` for viewer-oriented browsing
- no `sdata`-driven UI for:
  - coordinate systems
  - labels filtered by coordinate system
  - images filtered by coordinate system
  - linked tables for labels
  - channels for images
- no `Open SpatialData` action in the UI

### Architectural takeaway

The loading side is ready.

The missing part is mainly:

- widget UI
- `sdata` discovery helpers for the viewer flow
- state refresh logic

So `VW-03` should focus on orchestration and UI, not on new napari-loading
infrastructure.

## Reuse vs. Avoid

### Reuse

These pieces are already aligned with the target architecture:

- `HarpyAppState`
- `ViewerAdapter`
- `LayerBindingRegistry`
- `get_annotating_table_names(...)` from `_spatialdata.py`
- `get_table_metadata(...)` / `validate_table_binding(...)` when useful for
  labels-table linkage

### Avoid as the basis for the new viewer widget

These are still tied to viewer-scanning / legacy `napari-spatialdata` metadata:

- `SpatialDataViewerBinding`
- `get_spatialdata_label_options(viewer)`
- `get_spatialdata_image_options(viewer, ...)`
- `_get_spatialdata_*_from_viewer(...)`

Those helpers were useful for the existing task widgets, but the new `Viewer`
widget should work from the shared loaded `sdata`, not by scanning viewer
metadata.

## Recommended Implementation Slices

### Slice 1

Create the widget shell and wire it to shared app state.

Add:

- `src/napari_harpy/widgets/_viewer_widget.py`
- a `Viewer` widget command in `src/napari_harpy/napari.yaml`

Implement:

- a minimal `ViewerWidget(QWidget)`
- `self._app_state = get_or_create_app_state(napari_viewer)`
- subscription to `self._app_state.sdata_changed`
- a simple empty-state message when no `sdata` is loaded
- a minimal `refresh_from_sdata()` path

Also update:

- `src/napari_harpy/_interactive.py`

so `Interactive(...)` opens the new `Viewer` dock widget together with the
existing task widgets.

Recommended tests:

- widget can be instantiated
- widget is empty when no `sdata` is loaded
- widget refreshes when `sdata_changed` fires

Why start here:

- this gives us a visible dock widget quickly
- it proves the state wiring before we add more UI complexity
- it uses the already-available `Interactive(sdata)` path

### Slice 2

Add `sdata`-driven discovery helpers and coordinate-system-first UI.

Implement pure helper logic for:

- all coordinate systems available in the loaded `sdata`
- labels available in a selected coordinate system
- images available in a selected coordinate system
- correctly linked tables for a selected labels element
- channel names for a selected image

Recommended rule:

- keep these helpers `sdata`-based and pure
- if they are likely to be reused later by `VW-04` / `VW-05`, place them in
  `_spatialdata.py`
- but do not add new viewer-scanning helpers there

Implement UI state for:

- `Coordinate system`
- `Labels`
- `Linked table`
- `Image`
- `Display mode`

Recommended tests:

- coordinate systems are listed from the loaded `sdata`
- labels are filtered by coordinate system
- images are filtered by coordinate system
- linked tables are filtered to the selected labels element

### Slice 3

Implement the labels viewing path.

Add to the widget:

- `Show segmentation` action
- labels-specific refresh / activation behavior

Use:

- `self._app_state.viewer_adapter.ensure_labels_loaded(...)`
- `self._app_state.viewer_adapter.activate_layer(...)`

Recommended behavior:

- if the selected labels layer is already loaded in the selected coordinate
  system, just activate it
- otherwise load it

Recommended tests:

- clicking `Show segmentation` loads the expected labels layer
- repeated calls reuse the existing layer
- selection / activation is delegated to `ViewerAdapter`

### Slice 4

Implement stack image viewing.

Add:

- `Show image` action for `display_mode = stack`

Use:

- `ensure_image_loaded(..., mode="stack")`
- `activate_layer(...)`

Recommended tests:

- clicking `Show image` loads a stack image layer
- repeated calls reuse the existing stack layer
- the selected coordinate system is respected

### Slice 5

Implement overlay image viewing and channel-color UI.

Add:

- channel selection UI
- per-channel color controls
- `Show selected channels` action for `display_mode = overlay`

Use:

- `ensure_image_loaded(..., mode="overlay", channels=..., channel_colors=...)`

Recommended UI approach for the first version:

- a simple list of channel checkboxes
- one color control per selected channel
- no need to optimize for huge channel counts yet

Recommended tests:

- selected channels are passed to the adapter
- selected colors are passed to the adapter
- overlay mode reuses/removes layers as expected through the adapter

### Slice 6

Add `Open SpatialData`.

This is intentionally last because it is not needed to prove the core viewer
widget behavior while we already have `Interactive(sdata)`.

Implement:

- `Open SpatialData` button
- file selection
- `read_zarr(...)`
- `self._app_state.set_sdata(...)`
- clear user-facing error feedback if loading succeeds
- show a user-facing error message if loading fails

Recommended first-version scope:

- support zarr-backed `SpatialData` stores only
- do not try to solve every storage backend in `VW-03`

Recommended tests:

- monkeypatch file-dialog path selection
- monkeypatch `read_zarr(...)`
- verify `set_sdata(...)` is called
- verify the widget refreshes after loading
