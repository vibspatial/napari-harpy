# VW-02 Implementation Notes

## Recommendation

Do not start `VW-02` with multiplex image overlays.

Start with the smallest viewer-adapter slice that creates a clean foundation for
the later loading logic:

1. `LayerBindingRegistry`
2. `ViewerAdapter` lookup and activation helpers
3. `ensure_labels_loaded(...)`
4. stack image loading
5. overlay image loading
6. targeted real-viewer verification tests

## Why This Order

The current napari-facing logic is spread across
`src/napari_harpy/_spatialdata.py`, especially:

- `get_labels_layer(...)`
- `get_image_layer(...)`
- `set_active_layer(...)`
- `_get_loaded_spatialdata_layer(...)`

These are the easiest responsibilities to move first into a dedicated
`ViewerAdapter`.

Labels loading should come before image loading because:

- `Object Classification` will need it directly
- labels are simpler than multiplex images
- it gives immediate value without forcing channel-overlay decisions too early

Stack image loading should come before overlay mode because:

- it is simpler
- it helps stabilize the adapter API
- overlay mode is the most design-heavy part of `VW-02`

### Slice 2

Implement:

- `ensure_labels_loaded(sdata, label_name, coordinate_system)`

This is the first point where Harpy loads a labels element into napari without
depending on `napari-spatialdata`.

For this slice, keep layer metadata minimal:

- `element_name`
- `element_type="labels"`
- `coordinate_system`

The registry should remain the authoritative source of truth for layer
bindings.

### Slice 3

Implement stack image loading:

- `ensure_image_loaded(..., mode="stack")`

This should create one napari `Image` layer.

### Slice 4

Implement overlay image loading:

- selected channels
- one `Image` layer per selected channel
- additive blending
- per-channel colormap / color

### Slice 5

Add a small number of real-viewer integration tests once the core loading paths
are stable.

These should stay intentionally small and focus on napari behaviors that the
dummy-viewer unit tests do not fully validate:

- `Interactive(...)` plus dock-widget creation / reuse behavior
- real layer insertion and activation / selection behavior
- eventual `ensure_image_loaded(...)` overlay rendering paths

The goal of this slice is not to replace the fast unit tests. It is to add a
thin integration layer that catches napari-specific regressions in the most
important viewer flows.

## Architectural Note

Do not begin by moving all of `SpatialDataViewerBinding` out of
`_spatialdata.py` in one step.

That would be too much refactor at once.

Instead:

- copy the smallest napari-facing responsibilities into `ViewerAdapter`
- stabilize the adapter API
- switch callers over incrementally
