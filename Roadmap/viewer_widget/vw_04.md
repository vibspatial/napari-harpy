# VW-04 Implementation Notes

## Recommendation

Start `VW-04` by replacing viewer scanning with shared `sdata` wiring before
changing the feature-selection UI too much.

The current codebase already gives us a strong foundation for this migration:

- `HarpyAppState` already exposes the shared loaded `sdata`
- `HarpyAppState` already emits `sdata_changed`
- `ViewerWidget` already contains pure `sdata`-driven helpers for:
  - coordinate systems
  - labels in a coordinate system
  - images in a coordinate system
  - image channel names
- `FeatureExtractionController` already accepts:
  - `sdata`
  - `label_name`
  - `image_name`
  - `table_name`
  - `coordinate_system`
  - `feature_names`
  - `feature_key`
- `harpy.tb.add_feature_matrix(...)` already supports a `channels=` argument,
  and the Harpy feature-extraction controller/widget path now exposes it

That means `VW-04` is mainly a migration and orchestration task:

- replace viewer-scanning as the source of options
- subscribe the widget to shared `sdata`
- add explicit extraction-channel state
- preserve the existing controller-centric calculation flow
- finish the remaining legacy UX cleanup

UI rule for `VW-04`:

- follow the same coordinate-system-first pattern as `VW-03`
- users choose the coordinate system first
- segmentation masks, images, linked tables, and extraction channels are then
  derived from that coordinate-system context

## Current State Compared To The VW-04 Ticket

### Already available

- shared `HarpyAppState` per viewer
- shared loaded `sdata`
- `sdata_changed` signal
- `FeatureExtractionWidget` now subscribes to shared `sdata_changed`
- `FeatureExtractionWidget` now seeds itself from `self._app_state.sdata` on
  construction
- `FeatureExtractionWidget` now shows a clear empty state when no shared
  `SpatialData` is loaded
- a finished `Viewer` widget that is fully `sdata`-driven
- pure helper logic in `ViewerWidget` for:
  - coordinate systems
  - labels filtered by coordinate system
  - images filtered by coordinate system
  - image channel names
- `FeatureExtractionController` with stable bind/calculate flow
- `FeatureExtractionController` already validates:
  - segmentation presence
  - linked table presence
  - coordinate system presence
  - selected features
  - image requirement for intensity features

### Still legacy / not yet aligned with VW-04

- `ObjectClassificationWidget` still uses viewer-scanning patterns and will be
  handled in `VW-05`

### Architectural takeaway

The controller layer is already in decent shape.

The remaining work in `VW-04` is mainly final UX cleanup and alignment of the
selection/status messaging with the shared-`sdata` workflow.

So `VW-04` should focus on:

- cleanup of the last feature-extraction UX remnants from the migration
- final status and selection messaging
- tightening the shared-`sdata` story in the UI

not on reworking the core background-extraction pattern.

The target UI should therefore be:

1. choose coordinate system first
2. then choose segmentation mask
3. then choose image / extraction channels
4. then choose linked table and feature settings

## Reuse vs. Avoid

### Reuse

These pieces are already aligned with the target architecture:

- `HarpyAppState`
- `sdata_changed`
- `FeatureExtractionController`
- `get_annotating_table_names(...)`
- `get_table(...)`
- `validate_table_binding(...)`
- pure helper logic already implemented in `ViewerWidget`
- `harpy.tb.add_feature_matrix(..., channels=...)`

### Avoid as the basis for the new feature-extraction flow

These are still tied to viewer scanning / legacy metadata assumptions:

- `SpatialDataViewerBinding`
- `refresh_segmentation_masks()` as the primary source of options

Those may remain as temporary compatibility paths during the migration, but
they should not remain the main source of truth once `VW-04` is complete.

## Recommended Implementation Slices

### Slice 1

Wire the widget to shared app state and `sdata_changed`.

Status: completed

Implement:

- subscribe `FeatureExtractionWidget` to
  `self._app_state.sdata_changed.connect(...)`
- add a `refresh_from_sdata(sdata)` path
- when no `sdata` is loaded:
  - show a clear empty-state / status message
  - clear selector options
  - disable calculation
- seed the widget from `self._app_state.sdata` on construction

Keep for this slice:

- the existing controller
- the existing feature checkbox / output-key UI

Recommended tests:

- widget reflects `self._app_state.sdata` on construction
- widget refreshes when `sdata_changed` fires
- widget becomes empty/disabled when `sdata` is cleared

Delivered in this slice:

- `FeatureExtractionWidget` now subscribes to `self._app_state.sdata_changed`
- the widget now exposes `refresh_from_sdata(sdata)` and seeds itself from the
  shared app state on construction
- when shared `sdata` is `None`, the widget clears selectors, disables the
  rescan action, and shows a `No SpatialData Loaded` status card
- the legacy viewer-scanning flow remains in place, but it is now gated behind
  the presence of shared `sdata`

### Slice 2

Replace viewer-scanned labels/images/tables with `sdata`-driven option
discovery.

Status: completed

Implement:

- remove `SpatialDataViewerBinding` from the main selection flow
- replace:
  - labels discovery
  - image discovery
  - coordinate-system discovery
  with `sdata`-driven helpers
- prefer moving those pure helpers into `_spatialdata.py` if they will be
  reused by `VW-05`
- keep table lookup through:
  - `get_annotating_table_names(...)`
  - `validate_table_binding(...)`

Recommended behavior:

- the widget should now treat `self._app_state.sdata` as the source of truth
- viewer layer insertion/removal should no longer be required to discover
  selectable options

Recommended tests:

- segmentation choices come from the loaded `sdata`
- image choices come from the loaded `sdata`
- linked tables come from the selected labels element in `sdata`

Delivered in this slice:

- `FeatureExtractionWidget` now builds labels and images from shared
  `self._app_state.sdata` instead of `SpatialDataViewerBinding`
- option discovery no longer depends on viewer-linked layers being present
- linked tables continue to come from `get_annotating_table_names(...)` against
  the selected labels element in `sdata`
- new pure helper functions in `_spatialdata.py` now expose
  `SpatialDataLabelsOption` / `SpatialDataImageOption` discovery directly from
  a loaded `SpatialData` object

### Slice 3

Make the selection flow coordinate-system-first.

Status: completed

Implement:

- move `Coordinate system` to the top of the selection flow
- after coordinate-system selection:
  - show labels available in that coordinate system
  - show images available in that coordinate system
- keep `No image` as a valid explicit option
- preserve the current table validation behavior

Recommended behavior:

- coordinate system should no longer be derived only from the selected
  segmentation/image context
- labels and images should both be filtered by the chosen coordinate system

Recommended tests:

- coordinate systems are listed from the loaded `sdata`
- labels are filtered by coordinate system
- images are filtered by coordinate system
- the widget still supports a valid “no image selected” morphology-only path

Delivered in this slice:

- `FeatureExtractionWidget` now exposes `Coordinate system` at the top of the
  selection flow
- segmentation options are now filtered directly by the selected coordinate
  system
- image options are now filtered directly by the selected coordinate system
- table lookup remains tied to the selected segmentation, preserving the
  existing table validation behavior
- new pure helpers in `_spatialdata.py` now expose:
  - coordinate-system names from a loaded `SpatialData`
  - labels filtered by coordinate system
  - images filtered by coordinate system

### Slice 4

Add explicit extraction-channel selection local to the feature-extraction
widget.

Status: completed

Implement:

- expose channel names for the selected image
- add explicit extraction-channel UI
- keep this state local to the feature-extraction widget
- do not derive extraction channels from viewer display state
- ensure the selection is independent from the `Viewer` widget’s overlay
  channel choices

Recommended UX for a first version:

- if the selected image has no channel axis:
  - no extraction-channel UI is shown
- if it has a channel axis:
  - show a compact checkbox list
  - default to “all channels selected” or an equivalent explicit initial state
- if no image is selected:
  - no extraction-channel UI is shown

Recommended tests:

- channel names are listed from the selected image
- channel selection is local to the feature-extraction widget
- changing viewer overlay display state does not change extraction-channel state

Delivered in this slice:

- `FeatureExtractionWidget` now shows a local `Channels` selector row when the
  selected image exposes a channel axis
- extraction-channel names now come from a shared pure helper in
  `_spatialdata.py`, so both `ViewerWidget` and `FeatureExtractionWidget`
  resolve channels from the same `SpatialData` source of truth
- the extraction-channel UI remains hidden when no image is selected or when
  the selected image has no channel axis
- the feature-extraction widget keeps this channel state locally per selected
  image and defaults to all channels selected on first use
- extraction-channel state is independent from the `Viewer` widget overlay
  channel UI

### Slice 5

Extend the controller/job path to carry selected extraction channels.

Status: completed

Implement:

- extend `FeatureExtractionController.bind(...)` to accept selected channels
- store that channel selection in controller state
- extend `FeatureExtractionJob`
- pass `channels=` through `_run_feature_extraction_job(...)` into
  `harpy.tb.add_feature_matrix(...)`
- keep all existing validation rules intact

Recommended behavior:

- if no image is selected, `channels` should remain `None`
- if an image is selected and channels are explicitly selected, pass them
  through as the Harpy extraction channels

Recommended tests:

- controller bind stores selected channels
- calculate launches a job with the chosen channels
- morphology-only runs still work with `channels=None`

Delivered in this slice:

- `FeatureExtractionController.bind(...)` now accepts selected extraction
  channels and stores them in controller state
- `FeatureExtractionJob` now carries channels as part of the immutable worker
  payload
- `_run_feature_extraction_job(...)` now passes `channels=` into
  `harpy.tb.add_feature_matrix(...)`
- `FeatureExtractionWidget` now binds its locally selected extraction-channel
  names directly into the controller
- morphology-only runs still keep `channels=None`

### Slice 6

Clean up legacy viewer-scanning UX and finalize status messaging.

Status: completed

Implement:

- remove or repurpose `Rescan Viewer`
- remove now-obsolete viewer event refresh semantics from the primary path
- update selection/status text so it explains the shared-`sdata` workflow
- keep calculation/status messaging consistent with the new source of truth

Recommended end-state:

- the feature-extraction widget should read like a shared-state consumer, not a
  viewer-scanning tool
- any remaining viewer-scanning code should be only transitional or removed

Recommended tests:

- no user action depends on `Rescan Viewer`
- the widget stays synchronized through `sdata_changed`
- selection status remains informative after the migration

Delivered in this slice:

- the `Rescan Viewer` action has been removed from `FeatureExtractionWidget`
- the widget now reads as a shared-state consumer only and refreshes through
  `sdata_changed` / `refresh_from_sdata(...)`
- the empty-state message now explains that the form updates automatically from
  the shared Harpy state once a `SpatialData` object is loaded
- stale pre-migration messaging on the `Calculate` button tooltip has been
  replaced with copy that describes the current shared-`sdata` validation flow
