# VW-07 Implementation Notes

## Recommendation

Start `VW-07` by separating the notion of a **primary** labels layer from
**styled viewer variants** before adding new viewer UI controls.

The current codebase already gives us a strong foundation for this work:

- `ViewerWidget` already has a finished coordinate-system-first browsing flow
- each labels card already exposes the linked tables for a segmentation
- `ViewerAdapter` already owns labels-layer loading, lookup, activation, and
  layer binding
- `LayerBindingRegistry` is already the authoritative in-memory mapping from
  napari layers to Harpy element identity
- `ViewerStylingController` already knows how to color a labels layer from
  table-backed values and how to handle both categorical and continuous
  coloring patterns
- `ObjectClassificationWidget` already depends on `ViewerAdapter` and on the
  semantic `labels_layers_changed` signal for its live annotation layer

That means `VW-07` is mainly an adapter and viewer-UX orchestration task:

- reserve at most one canonical annotation-capable primary labels layer role
  for each `(sdata, labels element, coordinate system)` identity
- add separate styled labels-layer variants for viewer inspection
- keep object-classification styling independent from viewer-side inspection
  styling
- teach the viewer how to create or reuse table-colored labels overlays

The most important architectural rule for `VW-07` is:

- `ObjectClassificationWidget` should continue to care only about the primary
  labels layer
- table-colored viewer overlays should be separate styled variants and should
  not be treated as the annotation layer

Another important rule is:

- keep `_viewer_styling.py` focused on object-classification styling for the
  primary labels layer
- implement viewer-side overlay coloring through a separate viewer-specific
  styling path rather than trying to generalize the current
  `ViewerStylingController` in place

For the first implementation, I recommend we scope `VW-07` to:

- table `obs` columns
- `X[:, var_name]` coloring from the linked table

That still keeps the first version coherent while avoiding the broader
complexity of `.obsm[...]`-driven feature coloring.

## Current State Compared To The VW-07 Ticket

### Already available

- shared `HarpyAppState` per viewer
- shared loaded `sdata`
- finished `ViewerWidget` cards for images and segmentations
- linked-table discovery through `get_annotating_table_names(...)`
- `ViewerAdapter.ensure_labels_loaded(...)`
- `ViewerAdapter.get_loaded_labels_layer(...)`
- `ViewerAdapter.activate_layer(...)`
- `LayerBindingRegistry` and `LayerBinding`
- `ViewerStylingController` with:
  - direct labels-layer recoloring
  - categorical palette handling
  - continuous confidence-style coloring
  - labels-layer feature refresh
- existing object-classification color modes for:
  - `user_class`
  - `pred_class`
  - `pred_confidence`

### Current constraints we need to design around

- `ViewerAdapter.get_loaded_labels_layer(...)` currently assumes one logical
  labels layer per `(sdata, label_name, coordinate_system)`
- `labels_layers_changed` currently fires for any pickable labels-layer
  insert/remove/reorder event
- `ObjectClassificationWidget` listens to `labels_layers_changed` and treats it
  as live annotation-layer lifecycle
- `ViewerStylingController` currently mutates one bound labels layer in place,
  which is appropriate for object classification but not for persistent viewer
  overlays
- `ViewerWidget` does not yet expose any labels-coloring controls

### Architectural takeaway

The cleanest post-`VW-05` design is **not**:

- one labels layer plus a mutable `color_by` flag

The cleanest design is:

- zero or one **primary** labels layer used for annotation / interaction
- zero or more **styled** labels-layer variants used for viewer inspection

Styled overlays should be managed as identifiable reusable layer variants, not
as disposable one-off layers.

For example:

- if the user colors `blobs_labels` by `obs["cell_type"]`, the viewer should
  create one styled overlay for that style spec
- if the user later picks `obs["cell_type"]` again for `blobs_labels`, the
  viewer should reuse or update that same styled overlay rather than creating a
  duplicate
- if the user later picks `obs["tissue_zone"]` for `blobs_labels`, the viewer
  should create a second styled overlay because that is a different style spec

That means `VW-07` should extend the binding model with labels-layer role and
style metadata, and should narrow `labels_layers_changed` so
`ObjectClassificationWidget` only reacts to the primary annotation-capable
layer lifecycle.

For `VW-07`, the binding model should explicitly adopt the subtype / union
approach rather than continuing with one flat `LayerBinding` carrying many
type-specific optional fields. The intended direction is:

```python
@dataclass(frozen=True)
class BaseLayerBinding:
    layer: Layer
    element_name: str
    element_type: Literal["labels", "image"]
    coordinate_system: str | None
    sdata_id: int | None = None


@dataclass(frozen=True)
class LabelsStyleSpec:
    table_name: str
    source_kind: Literal["obs_column", "x_var"]
    value_key: str
    value_kind: Literal["categorical", "continuous"]


@dataclass(frozen=True)
class LabelsLayerBinding(BaseLayerBinding):
    element_type: Literal["labels"] = "labels"
    labels_role: Literal["primary", "styled"] = "primary"
    style_spec: LabelsStyleSpec | None = None


@dataclass(frozen=True)
class ImageLayerBinding(BaseLayerBinding):
    element_type: Literal["image"] = "image"
    image_display_mode: ImageDisplayMode | None = None
    channel_index: int | None = None
    channel_name: str | None = None


LayerBinding = LabelsLayerBinding | ImageLayerBinding
```

The registry should remain unified, but the binding types themselves should be
split this way so labels-specific and image-specific state stay explicit. In
other words: keep one registry, but let it store a `LayerBinding` union made
up of `LabelsLayerBinding` and `ImageLayerBinding`.

This is an important design choice for `VW-07`:

- do **not** solve this by moving the signal into `HarpyAppState`
- keep the labels-layer lifecycle signal on `ViewerAdapter`, because it is a
  viewer-layer concern rather than shared session state
- improve the design by narrowing and likely renaming the signal
  semantically, for example:
  - `primary_labels_layers_changed`
  - `annotation_labels_layers_changed`

## Reuse vs. Avoid

### Reuse

These pieces are already aligned with the target architecture:

- `ViewerWidget` labels cards and linked-table display
- `ViewerAdapter`
- `LayerBindingRegistry`
- `LayerBinding`
- `ViewerStylingController`
- `DirectLabelColormap`-based labels styling
- class-palette utilities in `_class_palette.py`
- `get_annotating_table_names(...)`
- `get_table(...)`
- `get_table_metadata(...)`
- categorical palette helpers in `_class_palette.py`

### Avoid as the basis for the new viewer-coloring flow

These would create confusion or fight the current architecture:

- treating every labels layer as equivalent from the adapter's perspective
- making `ObjectClassificationWidget` react to styled viewer overlays
- using a single mutable labels layer for both annotation and arbitrary viewer
  overlays
- reusing `_viewer_styling.py` as the main implementation path for viewer
  overlays, even though it is currently specialized to
  `user_class` / `pred_class` / `pred_confidence`
- encoding viewer styling as only a bare `color_by` string on the binding
- making the first version depend on `.obsm[...]` feature coloring before the
  `obs` / `X[:, var_name]` paths are stable

## Recommended First-Version Boundary

For `VW-07`, the first version should support:

- coloring labels layers from table-backed `obs` columns
- both categorical and continuous `obs` columns
- `bool` `obs` columns treated as categorical color sources
- exact binary `{0, 1}` integer `obs` columns treated as categorical while
  preserving `0` / `1` as the category values
- integer `obs` columns with only a small number of distinct values treated as
  categorical, while keeping those original integer labels as the categories
- other integer `obs` columns treated as continuous color sources
- plain string / object `obs` columns that are treated as temporary
  categorical color sources for viewer rendering, with a warning and default
  palette fallback
- coloring labels layers from `X[:, var_name]` of the linked table
- multiple styled overlays for the same segmentation, one per selected color
  source
- layer naming and binding metadata that make the chosen color source visible
  to the user
- categorical `obs` coloring that prefers stored `table.uns[f\"{column}_colors\"]`
  palettes when they are valid
- fallback categorical coloring that uses a deterministic Scanpy-style default
  palette when stored `_colors` entries are missing or invalid
- categorical palette interpretation that is based on the full original linked
  table category order, not on a queried / region-filtered subset

The first version should explicitly not try to solve:

- feature / gene coloring from `.obsm[...]`
- full legend widgets or rich palette editors
- table-category cleanup in the rendering path via
  `remove_unused_categories()`

Those can become a follow-up once the viewer-side styled-layer architecture is
stable.

## Recommended Implementation Slices

### Slice 1

Separate primary labels layers from styled viewer variants in the adapter and
binding model.

Status: not started

Implement:

- extend `LayerBinding` with explicit labels-layer role information, for
- replace the current flat `LayerBinding` with typed binding variants:
  - `BaseLayerBinding`
  - `LabelsLayerBinding`
  - `ImageLayerBinding`
- keep one unified `LayerBindingRegistry`, but let it store the typed binding
  union rather than one shape with many irrelevant optional fields
- in `LabelsLayerBinding`, store:
  - `labels_role="primary"` for annotation-capable layers
  - `labels_role="styled"` for viewer overlays
  - `style_spec: LabelsStyleSpec | None`
- in `ImageLayerBinding`, keep the image-specific fields:
  - `image_display_mode`
  - `channel_index`
  - `channel_name`
- for `X[:, var_name]` styling, store `var_name` as the stable source identity
  and resolve the corresponding `var` index at apply time rather than storing
  only an integer index
- keep the existing `ensure_labels_loaded(...)` path as the primary-layer path
- keep `get_loaded_labels_layer(...)` returning the primary layer by default
- add separate adapter lookup helpers for styled variants
- narrow the current labels-layer lifecycle signal so it is emitted only for
  primary annotation-capable labels layers
- as part of that narrowing, consider renaming the signal to something more
  explicit, for example:
  - `primary_labels_layers_changed`
  - `annotation_labels_layers_changed`
- keep that narrowed signal on `ViewerAdapter` rather than moving it into
  `HarpyAppState`

Recommended behavior:

- object classification should continue to treat the primary labels layer as
  the canonical annotation-capable layer for the selected
  `(sdata, labels element, coordinate system)` identity when such a layer is
  present
- styled viewer overlays should not be treated as annotation-layer lifecycle
  events

Recommended tests:

- primary labels lookups ignore styled variants
- styled labels lookups can find a specific style variant
- `labels_layers_changed` is emitted for primary labels layers but not for
  styled overlays
- object-classification behavior remains stable after styled overlays are added

### Slice 3

Add viewer UI controls for table-driven labels coloring.

Status: not started

Implement:

- extend the labels card in `ViewerWidget` with viewer-coloring controls
- keep one visible `Add / Update in viewer` action in the labels card
- branch internally from that one action based on the selected color source:
  - `None` -> add/update the primary labels layer
  - `Observations` / `Vars` -> add/update the styled overlay for the selected
    source
- expose:
  - linked table
  - one mutually exclusive color-source-kind selector, for example:
    - `None`
    - `Observations`
    - `Vars`
  - a source-specific searchable selector:
    - `Observation` when `Observations` is selected
    - `Var` when `Vars` is selected

  be long

  for example by including the table / value source in the styled layer name
- add a small action/status text so the user can tell what the current button
  press will do, for example:
  - `Action: add/update primary labels layer`
  - `Action: add/update colored overlay for obs["cell_type"]`

Recommended behavior:

- `Observations` and `Vars` should be mutually exclusive source kinds rather
  than separate independent toggles

  action explicit through the current source selection and status text

- when `Vars` is selected, the viewer should expose a searchable list of
  `var_names`
- users should be able to add more than one styled overlay for the same
  segmentation by choosing different columns
- if the user chooses the same style source again, the viewer should update or
  reuse the matching styled layer rather than duplicating it indefinitely

Recommended tests:

  available
- the `Observations` and `Vars` selector is mutually exclusive and drives the
  correct source-specific searchable control
- the single `Add / Update in viewer` button dispatches to the primary-layer

  compatible columns are available
- the current action/status text reflects whether the button will update the
  primary labels layer or a styled overlay
- the chosen style source is reflected in layer naming / status feedback

### Slice 4

Implement styled labels-layer loading, reuse, and coloring through the
adapter/controller path.

Status: not started

Implement:

- add a viewer-adapter path such as `ensure_styled_labels_loaded(...)`
- allow styled labels variants to coexist with an optional primary labels layer
- register styled variants in `LayerBindingRegistry` with:
  - labels role
  - style spec
- reuse or update an existing styled variant when the style spec matches
- apply coloring to the styled labels layer from the selected linked-table
  value source
- support:
  - categorical palette-based coloring
  - continuous colormap-based coloring
- implement continuous `X[:, var_name]` coloring by resolving `var_name` in the
  linked table and reading only the selected `X` column at apply time
- handle sparse `X` safely and avoid materializing the full matrix when only
  one `var_name` is selected
- for categorical `obs` coloring:
  - classify the source column as follows:
    - `bool` dtype -> categorical
    - exact binary integer `{0, 1}` -> categorical, preserving `0` / `1`
    - integer columns with only a small number of distinct values ->
      categorical, keeping those original integer labels as the categories
    - other integer columns -> continuous
    - string / object columns -> temporary in-memory categorical working
      series
  - first try to read `table.uns[f\"{column}_colors\"]`
  - if a valid stored `_colors` palette is found for a true categorical source,
    log that the viewer is using the stored palette
  - validate both palette length and color-token validity against the full
    original table category order
  - if the stored palette is missing or invalid, log a warning and fall back to
    a deterministic Scanpy-style default palette
  - build the category-to-color mapping from the full table column categories
    before applying it to the currently visible subset of instances
  - if stored `_colors` and the full-table categories disagree, do not try to
    partially salvage the palette; warn and use the fallback palette instead
  - do not call `remove_unused_categories()` in the rendering path, because
    that can break the alignment between stored category order and the `_colors`
    list
  - whenever the viewer coerces a non-categorical source into a categorical
    working representation for coloring, log a warning about that coercion
  - if an `obs` column is plain string / object dtype, derive a temporary
    in-memory categorical working series for coloring rather than mutating the
    shared table or changing the real `obs` column in place
  - in that string-conversion case, do not consult `table.uns[f\"{column}_colors\"]`;
    use the deterministic default palette directly to avoid subtle mismatches
    and confusing behavior
- keep object-classification `user_class` / `pred_class` / `pred_confidence`
  styling independent from these viewer-side overlays

Recommended behavior:

- the viewer can show:
  - zero or one primary labels layer
  - one styled layer colored by `obs["category_1"]`
  - another styled layer colored by `obs["category_2"]`
  - another styled layer colored by `X[:, "GeneA"]`
  all at the same time
- reusing the same style spec should update the existing styled layer instead
  of creating redundant duplicates
- viewer-side overlays should not interfere with object-classification color
  modes on the primary layer
- if a selected `obs` source is `bool`, exact binary `{0, 1}` integers, or an
  integer column with only a small number of distinct values, the viewer may
  color it categorically, but it should preserve those original integer labels
  rather than collapsing them through `bool`
- if a selected `obs` source is plain strings rather than categorical dtype,
  the viewer should still color it by deriving a temporary in-memory
  categorical working source, without writing that coercion back into
  `table.obs`, and warning about the coercion
- when a valid stored `_colors` palette is used, the viewer should log that it
  found and applied the stored palette

Recommended tests:

- a styled labels layer is created and registered with the expected binding
  role and style metadata
- repeated requests for the same style spec reuse or update the existing styled
  layer
- different style specs for the same segmentation create distinct styled
  overlays
- categorical and continuous `obs` columns both produce valid labels colormaps
- `bool`, exact binary `{0, 1}` integer, and other integer `obs` columns with
  only a small number of distinct values are categorized according to the
  rules above without losing their original integer labels
- string / object `obs` columns are warned about, converted into a temporary
  in-memory categorical working source, and colored without consulting stored
  `_colors`
- valid stored `_colors` palettes are logged when used; invalid stored
  palettes warn and fall back deterministically
- `X[:, var_name]` sources produce valid continuous labels colormaps
